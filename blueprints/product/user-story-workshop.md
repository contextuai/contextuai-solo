# User Story Mapping Workshop

A collaborative workshop for creating a comprehensive user story map that translates user needs into a structured, prioritized product backlog. This blueprint guides teams through persona identification, journey mapping, story writing, and backlog prioritization to ensure the product delivers real value to real users.

## Objective

Produce a complete user story map with well-defined personas, mapped user journeys, properly formatted user stories with acceptance criteria, and a prioritized backlog ready for sprint planning.

## Steps

### 1. Identify and Define Personas
Create 2-4 primary personas that represent your most important user types. For each persona, document their role, goals, frustrations, technical proficiency, and context of use. Go beyond demographics and capture their motivations and the jobs they are trying to accomplish. Use real customer data, interview notes, and support tickets to ground personas in reality rather than assumptions.

### 2. Map User Journeys
For each persona, trace the end-to-end journey they take to accomplish their primary goals using your product. Break each journey into high-level activities (the "backbone") and then into specific tasks within each activity. Arrange tasks left-to-right in chronological order and identify pain points, decision moments, and emotional highs and lows. This visual map becomes the foundation for generating stories.

### 3. Write User Stories
Convert each task in the journey map into a user story using the format: "As a [persona], I want to [action] so that [benefit]." Each story should be independent, negotiable, valuable, estimable, small, and testable (INVEST criteria). Avoid technical implementation details in the story itself. If a story feels too large, split it into smaller stories that can each be delivered independently and still provide value.

### 4. Define Acceptance Criteria
For each user story, write 3-5 acceptance criteria that describe the specific conditions that must be met for the story to be considered complete. Use the Given/When/Then format for clarity: "Given [context], when [action], then [expected result]." Include both happy path and edge case scenarios. Acceptance criteria should be testable and unambiguous so that any team member can verify whether the story is done.

### 5. Prioritize the Backlog
Organize stories into horizontal slices across the journey map, where each slice represents a release or iteration. The top slice is your minimum viable product: the smallest set of stories that delivers a complete, usable experience. Subsequent slices add richness, handle edge cases, and serve secondary personas. Use value versus effort as the primary lens for deciding what goes into each slice. Mark dependencies between stories so the team can plan execution order.

## Expected Output

A visual user story map organized by personas and journeys, a complete set of user stories in standard format with acceptance criteria, and a prioritized backlog divided into release slices with the MVP clearly delineated.

## Recommended Agents

- **Product Manager** — Facilitates the workshop, ensures stories align with product strategy, and makes prioritization decisions.
- **Market Researcher** — Provides customer insights, interview data, and behavioral evidence to validate personas and journeys.
- **CEO** — Ensures the user story map aligns with the company's vision and strategic objectives.

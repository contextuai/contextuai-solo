# ContextuAI Roadmap — H2 2026 → H1 2027

> **Thesis:** Agents were the 2025–26 trend and we shipped them (96 agents, crews, triggers, approvals). The next 12 months are about what comes *after* agents — and every major vector favors a local-first desktop platform. Solo is the lab; Enterprise is the product. Every feature below ships **Solo first**, then ports to Enterprise with auth/multi-tenancy/governance added.
>
> Created 2026-06-12. Companion to `MOONSHOT.md` (the March 2026 launch plan — Phase 1/2 largely shipped) and `docs/planning/` (current-gap specs SPEC-01…13, which are the **prerequisite baseline** for this roadmap).

## 1. Where the market is going (research, June 2026)

Five vectors define "after agents." Sources at the bottom.

| # | Vector | Evidence | Why it favors us |
|---|--------|----------|------------------|
| 1 | **Proactive AI** — assistants that act on signals, not prompts | Altman calls proactive AI "the next big phase after chatbots and agents"; "anticipatory intelligence" recurring across 2026–27 trend forecasts | Our approvals queue is the missing trust mechanism everyone else lacks: proactive suggestions are safe when a human gate already exists |
| 2 | **Memory** — persistent, structured, personal context | Mem0/Zep/4-tier memory architectures became a product category; OS vendors building "personal semantic index" of files/email/calendar | Local-first memory is a privacy moat: the assistant that knows your whole business *because nothing leaves the machine* |
| 3 | **Computer use / operator agents** | Went "from research demo to production category" in 15 months; OpenAI Codex controls desktops (Apr 2026); agentic browsers everywhere; Browser Use at ~89% WebVoyager | We already own the desktop. Coder mode's trust model (per-project trust + allowlist + visible runs) is exactly the pattern operator agents need |
| 4 | **Agent interoperability** — MCP / A2A / AP2-UCP | MCP "won" agent-to-tool (~97M downloads, all four labs); A2A for agent↔agent delegation (50+ partners, Agent Cards); agentic payments live (Alipay: 120M agent transactions/week; DBS×Mastercard first live agentic payment) | We're an MCP *client* today; becoming an MCP/A2A *server* makes every Solo install a node other agents can call — the OpenAI-compat play, one level up |
| 5 | **On-device AI goes mainstream + enterprise governance crisis** | Build 2026: Windows local agents on any GPU (DirectML, not NPU-only), Phi-Silica, Secure AI Enclave; 80% of Fortune 500 run agents and name observability/governance the bottleneck; eval users ship ~6x more to production | Hardware tailwind for Solo's GGUF stack; governance/evals is the Enterprise differentiator we can grow from Solo's approvals + analytics |

**One-line strategy:** *From agents to colleagues* — Solo's agents get memory (know you), initiative (act first), hands (operate software), and a passport (talk to other agents) — wrapped in the trust layer (approvals, logs, evals) that Enterprise buys.

## 2. Quarter-by-quarter plan

Numbers like SPEC-xx refer to `docs/planning/specs/`. Effort: S/M/L/XL. Each item lists its Enterprise port path.

### Q3 2026 (Jul–Sep) — "Foundation + Memory"

Goal: ship the trust/stability baseline, then the memory layer that everything later builds on.

| ID | Feature | What | Effort | Enterprise port |
|----|---------|------|--------|-----------------|
| F-0 | **Baseline: SPEC-01..08 + 12** | Sidecar auth, secrets at rest, lifecycle, atomic writes, SSRF guard, streaming durability, HF token, CSP. Non-negotiable before proactive/operator features (they raise the stakes on all of these) | M total | SEC items map to enterprise hardening review; mostly N/A (enterprise has auth already) |
| F-1 | **Unified Memory Layer ("Solo remembers")** | Local memory service: extract entities/preferences/decisions/episodes from chats, crew runs, and channel traffic into a structured store (SQLite + embeddings, 3 tiers: facts → episodes → persona). Injected into chat/crews alongside RAG. **Memory panel UI**: view / edit / delete / export every memory — visible memory = trust = differentiation vs. cloud assistants | L | Per-user + per-team memory collections in Mongo; admin policy on what's remembered; retention rules. This is a headline enterprise feature ("institutional memory") |
| F-2 | **GPU acceleration + local-runtime plurality** | Ship llama-cpp GPU variants (Vulkan/CUDA) with auto-detect + in-app benchmark ("your machine runs 9B at 32 tok/s"); add **Windows Foundry Local / Ollama / LM Studio as first-class inference backends** behind the existing dispatcher. Rides the Build-2026 wave of local AI on any GPU | M | Enterprise inference stays Bedrock/Anthropic; port = backend-agnostic dispatcher cleanup only |
| F-3 | Crew dry-run + KB staleness (SPEC-10, SPEC-11) + model preload (SPEC-09) | Already spec'd — close them out this quarter | M | Dry-run ports directly (Celery task flag); staleness ports to enterprise KB |

**Q3 demo moment:** "I told Solo my pricing once, in passing, three weeks ago. Today a customer asked on Telegram and the crew quoted it correctly — and I can open the Memory panel and see exactly where it learned that."

### Q4 2026 (Oct–Dec) — "Initiative (Pulse) + Digital Workers"

Goal: Solo stops waiting for prompts. This is the proactive-AI quarter — our answer to vector #1, and the next viral demo.

| ID | Feature | What | Effort | Enterprise port |
|----|---------|------|--------|-----------------|
| F-4 | **Pulse — proactive daily briefing** | A system crew runs on schedule + signals: new inbound across channels, KB folder changes, calendar (optional ICS), watched RSS/news keywords, crew run results → one morning briefing card stack ("3 things need you, 2 things I did, 1 thing I noticed"). Each card has actions (approve / open / dismiss). Tray notification + optional local TTS read-out | L | Team Pulse: per-user digests + org rollups; delivery via Slack/Teams/email. Strong daily-active driver in both products |
| F-5 | **Signal watchers + always-on mode** | Generalized watcher framework (the Reddit poller pattern, made pluggable): folder watcher, RSS/keyword watcher, channel-volume anomaly watcher. App can run headless in tray ("Solo is on duty"). All proactive *actions* route through the existing approvals queue — initiative without autonomy risk | M | Watchers → Agentic Event Bridge subscriptions (enterprise already has the bridge concept; this unifies the model) |
| F-6 | **Digital Workers ("hire, don't configure")** | Package agents+crew+KB+channels+schedule+watchers into a named role with a face: "Social Media Manager", "Research Analyst", "Support Rep", "Content Studio", "Bookkeeper-lite". Each has an onboarding interview (fills brand voice, KBs, channels), a KPI card (replies sent, approval rate, time saved), and a weekly self-review. Reframes the 96-agent library from *catalog* to *workforce* | L | THE enterprise narrative of 2026 ("agentic workforce"). Port = worker templates + department assignment + the governance pack (F-10) |
| F-7 | **Voice layer v1** | Local STT (whisper.cpp GGUF via existing model-hub infra) for push-to-talk chat + "read my Pulse" local TTS. No cloud audio, ever | M | Optional; enterprise voice goes through compliance review later |

**Q4 demo moment:** "I opened my laptop at 9am. Solo already triaged 14 overnight messages, drafted 6 replies (waiting in Approvals), noticed a competitor mention spike on Reddit, and read me the briefing out loud. I approved everything from the tray in 90 seconds."

## 3. Solo → Enterprise porting principles

1. **Solo is the lab.** Features prove demand and UX with zero infra cost; Enterprise pays for governance, multi-tenancy, and scale.
2. **Port pattern per feature:** auth (static admin → JWT/RBAC) · storage (SQLite-compat → Mongo, same repository interfaces — the Motor-compat layer was built for exactly this) · execution (in-process orchestrator → Celery queues) · governance hooks (every Solo feature must emit the events F-12's ledger needs, from day one, so the port is additive).
3. **Sequencing:** a feature ports one quarter after it stabilizes in Solo. Q3 memory → Q4 enterprise memory; Q4 workers/Pulse → Q1 enterprise workers; Q1 operator/registry → Q2 enterprise governance suite.
4. **What never ports down:** agent registry at org scale, compliance exports, A2A gateway, commerce — enterprise-only revenue lines.

## 4. Model strategy refresh (continuous thread)

- Qwen 3.5 (thinking+vision+256K) remains the local backbone; keep the catalog tracking each release within ~2 weeks (the gemma-gated incident showed catalog freshness is a reliability feature, not a nicety).
- **Vision becomes load-bearing in Q1 2027** (operator screenshots) — test VL pipelines in Q4, not when F-8 starts.
- Add small fast models for watcher/triage roles (0.8–2B class) so Pulse runs continuously without owning the machine; route per role via the existing dispatcher.
- Watch Phi-Silica/Windows ML as a zero-install inference path for low-end machines (F-2).

## 5. Risks & honest counterpoints

| Risk | Mitigation |
|------|------------|
| Proactive AI that's wrong is worse than reactive AI that's silent | Everything proactive lands in Approvals; Pulse cards are suggestions, not actions; per-watcher mute. Ship boring-but-right before clever |
| Operator mode is a security blast radius | Browser-only v1, allowlists, session recording, SPEC-01/05/12 as hard prerequisites; desktop-wide control deferred until trust tooling proves itself |
| Memory = privacy expectations spike | Memory panel with full view/edit/delete/export from v1; memory off-switch per scope; it's local — say so loudly |
| One developer + big roadmap | Each quarter has ONE flagship (F-1 → F-4/F-6 → F-8/F-9 → F-11/F-12); everything else is M-or-smaller. Cut scope, not the flagship. Community (marketplace) multiplies output |
| A2A/AP2 may shift under us | Server-mode behind experimental flags; commerce stays watch-list until protocols prove out |
